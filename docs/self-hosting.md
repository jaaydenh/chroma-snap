# Future self-hosting and migration notes

Chroma Snap is open source from day one, but the v1 supported product path remains hosted-first. The local development stack is intentionally file-backed so contributors can exercise capture, upload, diffing, review, and baseline promotion without PostgreSQL, S3, or a durable queue.

This document captures the migration path for future production self-hosting. It is not a claim that production-grade self-hosting is supported today.

## Current local MVP seams

| Concern | Local MVP | Future supported self-hosting target |
| --- | --- | --- |
| Metadata | JSON files under `.chroma-snap/server` | PostgreSQL migrations in `apps/api/migrations` |
| Artifacts | `FileArtifactStore` on local disk | S3-compatible private bucket with lifecycle policy |
| Queue | JSON queue records plus `QueueJobProcessor` helpers | Durable queue such as SQS, BullMQ, or Postgres-backed jobs |
| Auth | Dev auth, OIDC claim parsing, local admin secret | Verified GitHub Actions OIDC, GitHub App installation checks, OAuth sessions |
| Review UI | Static HTML renderer/server | Hosted app with production session/cookie handling |
| Observability | JSON-line metrics/log hooks | Metrics backend, log aggregation, tracing, alerting |

## PostgreSQL migration path

The intended relational schema lives in `apps/api/migrations`:

1. `001_initial_schema.sql`: repositories, sessions, artifacts, builds, baselines, comparisons, decisions, audit, and queue records.
2. `002_milestone_3_worker_foundations.sql`: retry metadata, baseline story/mode metadata, and retention policy settings.
3. `003_milestone_4_github_app.sql`: GitHub installations, webhooks, pull request metadata, refs, and check runs.
4. `004_milestone_6_baseline_promotion.sql`: baseline promotion context for approved PR reconciliation.
5. `005_milestone_7_private_beta_hardening.sql`: usage events, private-beta limits, and cleanup run history.

Future work should add a migration runner that:

- Records applied migration versions in PostgreSQL.
- Runs migrations transactionally before API or worker startup.
- Blocks startup when a required migration is missing.
- Provides backup and rollback guidance before destructive changes.

## Object storage migration path

A production artifact adapter should implement `ArtifactStore` from `packages/shared/src/artifact-storage.ts`:

- `putArtifact()` should write to a private bucket using object keys generated by the upload session.
- `readArtifact()` should be available to signed URL proxy endpoints and workers.
- `verifyArtifact()` should compare byte size and SHA-256 for upload finalization.
- `deleteArtifacts()` should be idempotent for cleanup jobs.
- `localPath()` is optional and only useful for file-backed diffing. S3-backed workers should download to a temp file or stream into the diff engine.

Artifacts must remain private by default. Public PR comments should link to the review UI, not raw object URLs.

## Durable queue migration path

The local queue contract is intentionally small:

- `diff-build`: process finalized manifests.
- `check-update`: publish or refresh GitHub Check state.
- `baseline-promotion`: reconcile approved PR snapshots on base-branch confirmation.
- `cleanup`: run artifact, comparison, and queue-job retention cleanup.

Future durable queue adapters should preserve idempotent handlers, attempt counts, exponential backoff, and dead-letter visibility. `createCleanupJobHandler()` in `apps/worker` is the first endpoint-driven cleanup handler for that queue model.

## Observability and limits

Milestone 7 emits JSON-line metrics for API requests, upload sessions, build finalization, worker diffs, worker errors, and cleanup. A self-hosted deployment should route these into a real backend and alert on at least:

- High `worker.error` rate.
- Queue jobs that exhaust retries.
- Builds stuck in queued or processing state.
- Cleanup failures or unexpected freed-byte spikes.
- `QUOTA_EXCEEDED` spikes for a repository.
- GitHub webhook signature failures.

Private-beta limits are currently configured through environment variables or server options. The migration `private_beta_limits` table is a future home for per-repository overrides managed through an admin UI or internal tooling.

## Security hardening still required

Before supported self-hosting, implement and validate:

- GitHub Actions OIDC JWKS signature verification.
- Required GitHub App installation and repository access checks on upload session creation.
- GitHub OAuth session handling for the review UI.
- CSRF protection for review decisions.
- TLS termination and secure cookie configuration.
- Secrets management for GitHub App private keys, webhook secrets, artifact signing keys, and admin secrets.
- Backups for PostgreSQL and protected baseline artifacts.

## Backup and restore outline

Until production adapters exist, local backups mean copying `.chroma-snap/server` and any generated reports. A future production restore runbook should include:

1. Pause API uploads and workers.
2. Snapshot PostgreSQL and object storage at a consistent point.
3. Restore metadata first, then verify referenced artifact object keys exist.
4. Run a dry-run cleanup to detect orphaned objects without deleting them.
5. Resume workers and watch queue depth, check-run updates, and error metrics.

## Upgrade notes

Open-source contributors should keep migration files append-only. Do not rewrite previously published migrations unless the repository is still in pre-release local development and all operators are told to reset. For hosted-first v1, document each milestone's schema and operational changes in `docs/private-beta-hardening.md`, `docs/data-model.md`, and the API migration README.
